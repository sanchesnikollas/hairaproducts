# Roadmap HAIRA: do estado atual a 100%

> **Para agentic workers:** REQUIRED SUB-SKILL: `superpowers:subagent-driven-development`. Steps usam checkbox (`- [ ]`). Despache subagents fresh por task; revise entre tasks. Use os agents existentes (`brand-onboarding`, `batch-processor`, `inci-enricher`, `pipeline-doctor`, `data-quality-auditor`, `frontend-reviewer`, `deploy-operator`) pra escala.

**Goal:** Levar o banco do estado atual (109 marcas com produtos / 19.901 produtos / 35.1% INCI) ao alvo de 100% definido em [2026-05-03-fechamento-coverage-693-marcas.md](2026-05-03-fechamento-coverage-693-marcas.md): **≥250 marcas com produtos individuais + ≥350 com catalog-stub via distribuidores**, qualidade vertical com **INCI ≥80%, price ≥90%, care_usage ≥60%, quarantined <50**, deploy em produção.

**Architecture:** 6 marcos sequenciais (M0 → M5). M0/M1 são pré-trabalho. M2/M3 são cobertura horizontal (largura). M4 é qualidade vertical (profundidade). M5 ship + manutenção. Backups obrigatórios antes de cada operação destrutiva.

**Tech Stack:** Python CLI (`haira scrape/labels/enrich/source-scrape`), pytest, scripts em `scripts/`, agents customizados em `.claude/agents/`, FastAPI + React frontend, Railway deploy.

**Baseline travado em 2026-05-12** (snapshot JSON em [data/baselines/2026-05-12-baseline.json](../../../data/baselines/2026-05-12-baseline.json) — todas as % comparadas contra esse snapshot, não contra "agora"):

### Gating M4

```sql
WITH coverage AS (
  SELECT
    100.0 * SUM(CASE WHEN inci_ingredients IS NOT NULL AND length(inci_ingredients) > 5 THEN 1 ELSE 0 END) / COUNT(*) AS inci_pct,
    100.0 * SUM(CASE WHEN price IS NOT NULL THEN 1 ELSE 0 END) / COUNT(*) AS price_pct,
    100.0 * SUM(CASE WHEN care_usage IS NOT NULL AND length(care_usage) > 0 THEN 1 ELSE 0 END) / COUNT(*) AS care_pct,
    SUM(CASE WHEN verification_status='quarantined' AND COALESCE(reason,'') != 'permanent' THEN 1 ELSE 0 END) AS quarantined_real
  FROM products
)
SELECT inci_pct >= 80 AND price_pct >= 90 AND care_pct >= 60 AND quarantined_real < 50 FROM coverage; -- = 1
```

## Manutenção pós-100% (contínua)

**Cadência:**
- **Semanal:** `data-quality-auditor` automático (cron Railway) → alerta se cobertura cair >2pp
- **Mensal:** re-scrape das 50 marcas com mais tráfego (preço/estoque mudam)
- **Trimestral:** `brand-onboarding` em wave pequena para marcas novas

**Sinais vermelhos:**
- INCI cobertura < 75% → blueprint quebrado, invocar `pipeline-doctor`
- Quarantined > 100 → re-rodar auto-resolve + ajustar regra
- Marca `status='active'` com 0 produtos por >30 dias → reclassificar `blocked`/`no_source`

## Riscos & mitigações

| Risco | Probabilidade | Mitigação |
|---|---|---|
| Anthropic API quota estourar (LLM fill em Q3) | média | `MAX_LLM_CALLS_PER_BRAND=50` já no .env; cair pra extração determinística |
| Cloudflare/anti-bot em distribuidor (M3) | alta | `curl_cffi` já lida; fallback Playwright; pacing `REQUEST_DELAY_SECONDS=3` |
| Banco crescer demais (231 MB → ~600 MB) | baixa | Railway PG paga por GB; migrar quando passar 1 GB |
| Wave de onboarding gera blueprint ruim que zera marca | média | Backup obrigatório por wave; `data-quality-auditor` revisa antes do commit |
| Frontend antigo de worktree confundir QA | alta (já aconteceu) | Sempre rodar do `main` exceto quando explicitamente testando branch — adicionar nota em CLAUDE.md |
| `discovered_total` se mexer durante o roadmap | média | Baseline travado em M0 (snapshot JSON); todas as % vs snapshot, não vs "agora" |
